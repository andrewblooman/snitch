Getting Started with Snitch

Snitch is an AppSec platform that aggregates security findings from multiple scanners, calculates per-application risk scores, and provides AI-powered remediation guidance.

1
Register an Application

Add your app in the Applications page. Provide a name, repository URL, and optional container image for scanning.

2
Run a Scan

Trigger a scan manually from the app detail page, or push results automatically from your CI/CD pipeline.

3
Review Findings

Triage findings by severity. Mark items as fixed, accepted, or false positive. The risk score updates automatically.

4
Get AI Remediation

Click "AI Remediation" on any finding to get a step-by-step fix plan generated by Anthropic Claude.

AI remediation requires an ANTHROPIC_API_KEY configured in Settings. Without it, a mock plan is returned — no error is raised.
Supported Scanners
Semgrep — SAST

Static analysis for code vulnerabilities. Runs on cloned repositories. Findings stored as finding_type=SAST.

Trivy — SCA

Software composition analysis for dependency CVEs. Stored as finding_type=SCA.

Grype — Container

Container image vulnerability scanning. Set a container_image on your application to enable. Stored as finding_type=container.

Checkov — IaC

Infrastructure-as-code scanning for Terraform, CloudFormation, ARM, and Bicep. Stored as finding_type=IaC.

Gitleaks — Secrets

Detects leaked credentials and secrets in source code. Secret values are masked before storage. Stored as finding_type=secrets.

Govulncheck — Go

Go-specific vulnerability scanning using the official Go vulnerability database. Stored as finding_type=SCA.

Risk Score Formula

Each application's risk score is calculated from open findings only:

formula
score = (critical × 25) + (high × 10) + (medium × 3) + (low × 1)
score = min(score, 100)

Risk levels:
  0          → info
  1  – 24   → low
  25 – 49   → medium
  50 – 74   → high
  75 – 100  → critical
Marking findings as fixed, accepted, or false_positive immediately removes them from the score calculation.
GitHub Actions Integration

Push scan results directly to Snitch from your CI/CD pipeline using the POST /api/v1/cicd-scans/push endpoint. Results are normalised and deduplicated automatically.

You need your application's UUID from the Snitch Applications page, and the Snitch base URL (e.g. http://snitch.internal). Store these as GitHub Actions secrets.
Required Secrets

Add the following secrets to your GitHub repository (Settings → Secrets and variables → Actions):

secretsGitHub → Settings → Secrets
SNITCH_URL          # e.g. https://snitch.example.com
SNITCH_APP_ID       # UUID of the application in Snitch
Complete Workflow — All Scanners

The workflow below runs all supported scanners and pushes each result set to Snitch. Adjust steps to match your project's language and infrastructure.

yaml.github/workflows/snitch-scan.yml
name: Snitch Security Scan

on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]
  schedule:
    - cron: '0 2 * * 1'  # Weekly Monday 02:00 UTC

jobs:
  snitch:
    runs-on: ubuntu-latest
    permissions:
      contents: read

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      # ── Semgrep (SAST) ────────────────────────────────────────────
      - name: Run Semgrep
        uses: returntocorp/semgrep-action@v1
        with:
          config: p/default
          generateSarif: "false"
        env:
          SEMGREP_RULES: p/default

      - name: Run Semgrep (JSON output)
        run: |
          pip install semgrep --quiet
          semgrep --config p/default --json --output semgrep-results.json . || true

      - name: Push Semgrep results to Snitch
        run: |
          curl -sf -X POST ${{ secrets.SNITCH_URL }}/api/v1/cicd-scans/push \
            -H "Content-Type: application/json" \
            -d "{
              \"application_id\": \"${{ secrets.SNITCH_APP_ID }}\",
              \"scan_type\": \"semgrep\",
              \"branch\": \"${{ github.ref_name }}\",
              \"commit_sha\": \"${{ github.sha }}\",
              \"results\": $(cat semgrep-results.json)
            }"

      # ── Trivy (SCA) ───────────────────────────────────────────────
      - name: Run Trivy (filesystem / SCA)
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: fs
          format: json
          output: trivy-results.json
          exit-code: '0'

      - name: Push Trivy results to Snitch
        run: |
          curl -sf -X POST ${{ secrets.SNITCH_URL }}/api/v1/cicd-scans/push \
            -H "Content-Type: application/json" \
            -d "{
              \"application_id\": \"${{ secrets.SNITCH_APP_ID }}\",
              \"scan_type\": \"trivy\",
              \"branch\": \"${{ github.ref_name }}\",
              \"commit_sha\": \"${{ github.sha }}\",
              \"results\": $(cat trivy-results.json)
            }"

      # ── Checkov (IaC) ─────────────────────────────────────────────
      - name: Run Checkov
        uses: bridgecrewio/checkov-action@master
        with:
          directory: .
          framework: terraform,cloudformation,arm,bicep
          output_format: json
          output_file_path: checkov-results.json
          soft_fail: true

      - name: Push Checkov results to Snitch
        run: |
          curl -sf -X POST ${{ secrets.SNITCH_URL }}/api/v1/cicd-scans/push \
            -H "Content-Type: application/json" \
            -d "{
              \"application_id\": \"${{ secrets.SNITCH_APP_ID }}\",
              \"scan_type\": \"checkov\",
              \"branch\": \"${{ github.ref_name }}\",
              \"commit_sha\": \"${{ github.sha }}\",
              \"results\": $(cat checkov-results.json)
            }"

      # ── Gitleaks (Secrets) ────────────────────────────────────────
      - name: Run Gitleaks
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          GITLEAKS_LICENSE: ${{ secrets.GITLEAKS_LICENSE }}
        with:
          args: --report-format json --report-path gitleaks-results.json --exit-code 0

      - name: Push Gitleaks results to Snitch
        if: always()
        run: |
          curl -sf -X POST ${{ secrets.SNITCH_URL }}/api/v1/cicd-scans/push \
            -H "Content-Type: application/json" \
            -d "{
              \"application_id\": \"${{ secrets.SNITCH_APP_ID }}\",
              \"scan_type\": \"gitleaks\",
              \"branch\": \"${{ github.ref_name }}\",
              \"commit_sha\": \"${{ github.sha }}\",
              \"results\": $(cat gitleaks-results.json)
            }"
Container Scanning with Grype

To scan a container image, add the Grype step and ensure the container_image field is set on your Snitch application. Grype expects a JSON output from the --output json flag.

yaml.github/workflows/snitch-scan.yml (Grype snippet)
      # ── Grype (Container) ─────────────────────────────────────────
      - name: Install Grype
        run: curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin

      - name: Run Grype
        run: grype ${{ secrets.CONTAINER_IMAGE }} --output json --file grype-results.json || true

      - name: Push Grype results to Snitch
        run: |
          curl -sf -X POST ${{ secrets.SNITCH_URL }}/api/v1/cicd-scans/push \
            -H "Content-Type: application/json" \
            -d "{
              \"application_id\": \"${{ secrets.SNITCH_APP_ID }}\",
              \"scan_type\": \"grype\",
              \"branch\": \"${{ github.ref_name }}\",
              \"commit_sha\": \"${{ github.sha }}\",
              \"results\": $(cat grype-results.json)
            }"
Minimal Semgrep-Only Workflow

If you only need SAST scanning, this minimal workflow is a good starting point:

yaml.github/workflows/snitch-sast.yml
name: Snitch SAST
on: [push, pull_request]

jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Semgrep scan
        run: |
          pip install semgrep --quiet
          semgrep --config auto --json --output semgrep-results.json . || true

      - name: Push to Snitch
        run: |
          curl -sf -X POST ${{ secrets.SNITCH_URL }}/api/v1/cicd-scans/push \
            -H "Content-Type: application/json" \
            -d "{
              \"application_id\": \"${{ secrets.SNITCH_APP_ID }}\",
              \"scan_type\": \"semgrep\",
              \"branch\": \"${{ github.ref_name }}\",
              \"commit_sha\": \"${{ github.sha }}\",
              \"results\": $(cat semgrep-results.json)
            }"
Always use || true (or exit-code: '0') on scanner steps so that findings don't fail the workflow before the push step runs. Let Snitch policies decide whether to block a build.
Platform Walkthrough

A guide to the main Snitch features and how to use them effectively.

Applications

The Applications page lists all registered apps sorted by risk score (highest first). Click any column header to re-sort. Click an application row to open its detail page.

When creating an application you can optionally provide a repository URL (for on-demand cloning during scans) and a container image (for Grype container scanning). Leave the container image blank to skip that step.

Findings

Findings are organised by severity: critical → high → medium → low → info. Each finding shows the scanner, file path, rule ID, and CVE (for dependency findings).

Status transitions:

status flow
open  →  fixed          (vulnerability has been remediated)
open  →  accepted       (risk accepted, won't fix)
open  →  false_positive (not a real vulnerability)

Only open findings contribute to the risk score.

Policies

Policies let you define automated rules that run after every scan. A policy specifies:

  • Minimum severity — findings below this threshold are ignored by the policy
  • Scan types — which scanner results to evaluate (sast, sca, container, secrets, iac)
  • Rule blocklist / allowlist — specific Semgrep/Checkov rule IDs to include or exclude
  • Action — inform (log only) or block (marks the scan as blocked)
Use block policies in combination with CI/CD to gate deployments: check the scan result status via GET /api/v1/cicd-scans/{id} and fail the pipeline if blocked=true.
Rules Library

The Rules page provides a searchable catalog of ~37 built-in detection rules across Semgrep (SAST), Checkov (IaC), and Gitleaks (secrets). Each rule shows its ID, severity, and description. Use these IDs when configuring policy blocklists and allowlists.

Secrets

The Secrets page manages custom SecretPattern rows. Active patterns are loaded at scan time and passed to Gitleaks as a TOML config file. Raw secret values are never stored — only the last 4 characters are retained (****{last4}).

Reports

The Reports page provides aggregate views: findings by severity, risk score trends, scanner coverage, and top vulnerable applications. Data is computed from the live database and requires no separate ETL.

Repositories

If a GitHub Token is configured in Settings, the Repositories page can sync GitHub code scanning alerts directly into Snitch findings, and create fix branches or pull requests for findings with AI remediation plans.

Settings

The Settings page allows configuration of platform integrations (GitHub Token, Anthropic API Key) and scan defaults. Token presence flags are stored in localStorage — raw secrets are never persisted client-side.

Dark / Light Mode

The theme toggle is available in the header user menu (top right). Your preference is persisted in localStorage.theme and applied on every page load without flash.

AI Remediation

On any finding's detail view, click "Get AI Remediation" to request a step-by-step fix plan. The plan is generated by Anthropic Claude (model: claude-3-5-sonnet-20241022) with extended thinking enabled. Plans are cached in the database and can be retrieved via GET /api/v1/remediations/{finding_id}.

REST API Reference

All endpoints are prefixed /api/v1/. The interactive Swagger UI is available at /docs.

Applications
MethodPathDescription
GET/api/v1/applicationsList all applications (paginated, sortable)
POST/api/v1/applicationsCreate a new application
GET/api/v1/applications/{id}Get application detail with findings summary
DELETE/api/v1/applications/{id}Delete application and all its findings
POST/api/v1/applications/{id}/scanTrigger an async scan (Celery task)
Findings
MethodPathDescription
GET/api/v1/findingsList findings with filters (severity, status, scanner, application_id)
GET/api/v1/findings/{id}Get a single finding
POST/api/v1/findings/{id}/statusUpdate finding status (open/fixed/accepted/false_positive)
CI/CD Scan Push
MethodPathDescription
POST/api/v1/cicd-scans/pushIngest scanner JSON output from CI/CD pipeline
GET/api/v1/cicd-scansList CI/CD scan records (filterable by application, status, scan_type)
GET/api/v1/cicd-scans/{id}Get a specific CI/CD scan record (check blocked status)
Remediations
MethodPathDescription
POST/api/v1/remediations/{finding_id}Generate (or retrieve cached) AI remediation plan
GET/api/v1/remediations/{finding_id}Get existing remediation plan for a finding

Example: Check if a Scan was Blocked
bashcurl (use in CI/CD gating)
SCAN_ID="<id from push response>"

BLOCKED=$(curl -sf http://snitch.example.com/api/v1/cicd-scans/$SCAN_ID \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['blocked'])")

if [ "$BLOCKED" = "True" ]; then
  echo "❌ Snitch policy blocked this build"
  exit 1
fi
Push Payload Schema
jsonPOST /api/v1/cicd-scans/push
{
  "application_id": "uuid",          // required
  "scan_type":      "semgrep",         // semgrep | trivy | grype | checkov | gitleaks
  "branch":         "main",            // optional
  "commit_sha":     "abc123",          // optional
  "results":        { /* raw JSON output from the scanner */ }
}
Snitch auto-detects the scanner format from the JSON structure, so you can also use "scan_type": "auto" and let the normaliser figure it out.
