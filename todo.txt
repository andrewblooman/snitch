  Top picks (high value, low effort):

  1. Settings Page - Add a page for general admin and configuration
  2. Notifications/alerting — Slack webhooks or email when critical findings appear or SLA deadlines breach. Without push alerts, engineers only see findings when they log in. A notifications.py service +
  Celery Beat task would cover this.
  3. SLA tracking — Finding already has created_at/last_seen_at. Adding a sla_deadline field (auto-set by severity on upsert) enables breach reporting and feeds into notifications. Compliance frameworks (SOC2,
  ISO 27001) typically require this.
  4. SBOM export — Trivy already collects package data. Exposing a GET /applications/{id}/sbom?format=cyclonedx endpoint would satisfy an increasing regulatory requirement (US EO, EU CRA) with minimal new code.

  ---
  Medium effort, high value:

  5. Policy engine / CI gate — a /policy-check endpoint that CI/CD jobs can query post-upload to get a pass/fail decision. You have the S3/SQS ingestion pipeline but no feedback loop back to CI.
  6. Waivers/suppression workflow — status can currently be set to accepted with no audit trail or approval gate. A Waiver model with requester → approver flow would be needed for any regulated environment.
  7. Add AI Prompt configuration page for remediation

  ---
  Longer term:

  8. DAST (OWASP ZAP) — covers runtime vulnerabilities SAST/SCA miss, but requires a live target URL and is operationally heavier.
  9. Developer self-service portal — simplified view for devs with one-click remediation PRs. The AI remediation + GitHub PR creation are already built; this is mostly UX.
